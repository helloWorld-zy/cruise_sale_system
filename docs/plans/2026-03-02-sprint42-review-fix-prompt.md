# Sprint 4.2 Code Review 问题修复执行提示词（中文）

> 用途：将本次 code review 发现的问题系统化修复，确保可回归、可验收、可复审。
> 适用对象：代码代理/开发同学（后端 Go + 前端 Nuxt/Vue + uni-app）。

---

## 可直接使用的提示词

你现在是本项目的高级修复工程师，请基于仓库根目录 `TODO.md` 的审计清单执行一次完整修复。你的目标不是“局部改代码”，而是让 Sprint 4.2 达到可复审通过状态。

### 1. 修复总目标

1. 严格按 `TODO.md` 的优先级修复：`P0 -> P1 -> P2`。
2. 每完成一条，必须同步补测试与必要文档说明。
3. 修复后要能给出“可核验证据”：代码路径、测试结果、接口联调结果。
4. 每批修复后，立即更新 `TODO.md` 勾选状态并附证据，避免最终集中返工。
5. 保证不引入回归：关键链路（下单、支付、关单、导出、权限）必须有回归测试。

### 2. 强制执行规则

1. 任何 P0 未完成，不得宣称“本次修复完成”。
2. 涉及金额、库存、订单状态的改动，必须使用事务与幂等控制。
3. 涉及认证与回调（支付宝/支付回调），必须先做签名校验与防伪造校验。
4. 涉及导出功能，必须做权限校验、条数上限、CSV/公式注入防护。
5. 涉及前端页面，必须落地真实 API 调用，且具备 `loading/error/empty` 三态。
6. 禁止保留“占位实现”与“空函数体假逻辑”。

### 3. 修复范围（按 TODO.md 执行）

#### P0（必须先完成）

1. 前端实联补齐：
   - Admin 缺失页（员工管理、店铺设置、通知模板等）补齐并接后端真实 API。
   - Miniapp 舱位列表去硬编码（`frontend/miniapp/pages/cabin/list.vue`、`frontend/miniapp/components/CabinCard.vue`）。
   - 所有相关页面补三态处理（loading/error/empty）。

2. 支付宝登录与账号绑定安全：
   - `AlipayLogin` 增加签名验签与 UID 防伪造。
   - `BindAccount` 增加“唯一绑定校验 + 绑定前身份确认”。

3. 订单超时关单一致性：
   - `CloseExpiredOrders` 使用事务封装“关单 + 释放库存 + 日志”。
   - 增加并发防重与幂等处理。
   - 覆盖并发与失败回滚测试。

4. 订单状态日志落地：
   - 每次状态变更强制写 `OrderStatusLog(operator_id, remark)`。
   - 状态变更与日志写入同一事务。
   - 统一从状态机入口变更状态，不允许绕过 `CanTransitionTo`。

5. 导出安全：
   - 替换 `Excel content placeholder` 为真实导出。
   - 导出权限控制 + 数据量上限 + CSV 注入防护。

6. 通知模板安全：
   - 模板渲染改为 `text/template`。
   - 增加变量白名单与异常模板测试（防 SSTI）。

7. 数据看板增强落地：
   - `Trend`、`CabinHotnessRanking`、`InventoryOverview`、`PageViewStats` 实现真实查询。
   - 增加必要索引与性能相关测试/说明。

#### P1（高优先）

1. 批量操作安全：数量上限、事务回滚、权限与审计日志测试。
2. 库存预警：补 `AvailableWithAlert` 方法与阈值边界测试。
3. 批量日期范围定价 API：补 handler/router + 接口测试。
4. 对账日报幂等：按 `date` 防重复生成 + 并发测试。
5. 员工角色与 Casbin 同步：角色变更同步 policy/grouping，并写操作日志。
6. ShopInfo 单行约束：数据库层 + 服务层双重约束。
7. 库存预警通知防重复：增加去重键、时间窗口与重复扫描测试。

#### P2（完整性与质量）

1. 退款边界测试补齐（恰好 30/7/0 天）。
2. 清理空实现与伪覆盖。
3. 按 `docs/plans/2026-02-22-sprint04.2.md` 做 Part A-E / 27 Tasks 对照复核。
4. 前端目录整理：将 `frontend/admin/pages/dashboard`、`frontend/admin/pages/finance` 迁移到 `frontend/admin/app/pages`，并验证路由与冒烟测试。

### 4. 交付格式要求（每轮修复后输出）

请按以下结构输出，不要省略：

1. 本轮修复项（对应 TODO 勾选项）
2. 代码变更清单（文件路径 + 关键改动点）
3. 测试结果（执行命令 + 通过/失败 + 失败原因）
4. 风险与回滚点（如有）
5. TODO 勾选更新建议

### 5. 完成门禁（DoD）

仅当以下全部满足，才可声明“修复完成，可复审”：

1. `TODO.md` 中所有 P0 项已完成并可证据化。
2. 相关新增/修改功能均有测试覆盖（单测优先，必要时集成/E2E）。
3. 前后端关键路径已联调通过（下单、支付、关单、导出、权限）。
4. 无占位实现、无空函数假逻辑、无硬编码假数据页面。
